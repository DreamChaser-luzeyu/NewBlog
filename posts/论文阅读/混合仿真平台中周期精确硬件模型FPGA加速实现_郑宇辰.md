# 摘要

在集成电路设计过程中，仿真是至关重要的一环，随着集成电路规模的不断扩大，带来了设计复杂度和仿真难度的指数级增长，大规模的电路仿真会耗费大量的时间，因此，在激烈的市场竞争环境下，通过改进仿真技术，缩短设计周期和降低迭代次数显得尤为重要。本论文针对Verilog周期精确硬件模型加速实现原理进行深入研究，在FPGA上自动部署开发工具以实现仿真加速功能，同时将实现仿真加速的模型在混合仿真平台中实现，进行大规模系统级别的仿真验证。

本文首先介绍周期精确硬件模型的定义并分析相应的 LI-BDN 电路实现过程，进行了混合仿真平台中的硬件仿真模型及硬件接口的方案设计。根据方案设计，对 Yosys 开源编译器的修改，实现了周期精确硬件模型和相应硬件接口的自动化生成，最终将自动化生成的硬件电路在 FPGA 中综合实现。

在已实现的硬件仿真模型和硬件接口的基础上，实现了两种典型的混合仿真平台，基于Zynq的嵌入式混合仿真平台和基于QEMU的混合仿真平台。基于Zynq平台特点设计嵌入式混合仿真平台方案，并依照方案进行了Zynq中的软硬件通信、软件接口的实现，最后根据软件接口的实现设计了软件激励方案。基于QEMU的混合仿真平台是在已有的基于Zynq的嵌入式混合仿真平台上，结合QEMU软件仿真器，在Zynq上设计与QEMU交互的通信模块，并实现信号转换。这两个平台都利用了硬件仿真加速，实现了高效率的数字电路仿真，展示了混合仿真技术在复杂电路设计中的实际应用价值。

在本文的最后，对各个部分的具体设计进行了测试验证。使用开源 FFT IP 核进行测试，利用改进的 Yosys 编译器进行处理，对比处理前后的电路以及关键波形，确认周期精确硬件模型的自动化实现的正确性。根据软件激励的实现要求，对 FFT 的电路模型进行软件仿真模型设计，通过输出的结果，验证了软件仿真模型设计的正确性，并评估了仿真加速效果。最后对基于 QEMU 的混合仿真平台进行测试，通过系统级仿真，验证了该仿真平台的正确性。

# Abstract

In the process of integrated circuit design, simulation is a crucial step. As the scale of integrated circuits continues to expand, it brings about an exponential increase in design complexity and simulation difficulty. Large-scale circuit simulation can consume a significant amount of time. Therefore, in a fiercely competitive market environment, it is particularly important to shorten the design cycle and reduce the number of iterations by improving simulation technology. This thesis conducts an in-depth study of the principles of accelerating the implementation of Verilog cycle-accurate hardware models, and automatically deploys development tools on FPGA to implement simulation acceleration functions. Simultaneously, the model for realizing simulation acceleration is implemented on a hybrid simulation platform for large-scale system-level simulation verification.

This thesis first introduces the definition of the periodic precise hardware model and analyzes the corresponding implementation process of the LI-BDN circuit. The hardware simulation model and hardware interface scheme design in the hybrid simulation platform are carried out. According to the scheme design, modifications were made to the Yosys open-source compiler to achieve automatic generation of periodic accurate hardware models and corresponding hardware interfaces. Finally, the automatically generated hardware circuits were comprehensively implemented in FPGA.

Based on the realized hardware simulation model and hardware interface, two typical hybrid simulation platforms are implemented: the embedded hybrid simulation platform based on Zynq and the hybrid simulation platform based on QEMU. The embedded hybrid simulation platform scheme is designed based on the features of the Zynq platform, and software-hardware communication in Zynq and the implementation of the software interface are carried out according to the scheme. Finally, a software stimulus scheme is designed based on the implementation of the software interface. The hybrid simulation platform based on QEMU is integrated with the QEMU software simulator on the existing embedded hybrid simulation platform based on Zynq, and a communication module interacting with QEMU is designed on Zynq, implementing signal conversion. Both platforms take advantage of hardware simulation acceleration to achieve efficient digital circuit simulation, demonstrating the practical application value of hybrid simulation technology in complex circuit design.

In the end, the specific design of each part is tested and verified. The open-source FFT IP core is used for testing, the improved Yosys compiler is used for processing, and the circuits before and after processing and key waveforms are compared to confirm the correctness of the automatic implementation of cycle-accurate hardware models.

According to the implementation requirements of software stimulation, the FFT circuit model is designed for software simulation models, and the correctness of the design of software simulation models is verified through output results, and the effect of simulation acceleration is evaluated. Finally, the hybrid simulation platform based on QEMU is tested, and the correctness of this simulation platform is verified through system-level simulation.

Keywords: hybrid simulation, hardware emulation, cycle accurate, FPGA, Yosys

# 第1章 绪论

# 1.1 课题背景及研究的目的和意义

随着集成电路行业发展，集成电路规模越来越大，片上系统（System on Chip）近几年得到了广泛应用，SoC 是一个针对专用目标设计功能的集成电路，其中包含完整系统并能运行软件的全部内容[1,2]。随着技术发展以及需求增长，SoC 性能快速上升，近几年晶体管数量最多的 SoC 芯片已经达到百亿级别规模。随着芯片规模越来越大，芯片的复杂程度成指数上升。

设计周期和资源是集成电路项目的关键因素。在芯片设计阶段，仿真是至关重要的一环，通过电路仿真，工程师可以尽早发现和修复设计中的问题，从而避免多次迭代和额外的开发成本，特别是在高度竞争的市场中显得尤为重要。但是随着数字芯片复杂度提升，在RTL级别进行集成电路架构研究和仿真验证难度越来越大，主要受限于Verilog等RTL语言的仿真速度，一个大规模集成电路的仿真甚至可能需要在大型服务器上运行一个月。一个大规模的项目可能因为受限于TTM（Time to Market）[3]，在流片前能跑的测试程序有限，甚至可能无法完全达到验证目标。

# 1.1.1 现有仿真技术

为了解决开发过程中各种各样的问题，目前市场上现有不同仿真技术，适用于不同情况。根据精度和细节级别，仿真技术可以被划分为指令级别（Instruction Level）、事务级别（Transaction Level）和周期精确级别（Cycle Accurate Level）[4]。

指令级别的仿真主要用于模拟处理器的行为。在这个级别的仿真中，每一条处理器指令都被单独地模拟[5]。这种技术可以帮助设计人员理解处理器的行为和性能，但是它不包括处理器外部的系统级别的行为。它通常用于处理器设计和优化，以及软件早期开发。

事务级别的仿真关注的是系统级别的行为，而不是单个周期内的细节[6]。事务通常代表了一系列的操作，例如读写操作或数据传输。在这种级别的仿真中，设计人员可以更快速地模拟和验证大型系统，但是可能会错过一些关键的低级细节。这种技术通常用于系统架构设计和早期验证阶段。

周期精确级别仿真技术在每一个时钟周期内模拟电路的行为，可以捕捉到设计中的所有细节[7]。它提供了最高级别的精度，因此通常用于验证关键模块的功能和性能。然而，这种级别的仿真通常需要大量的计算资源和时间，所以并不适合大规模或复杂电路的全系统仿真。

可以看到不同精度的仿真模型有不同的特点，在芯片开发过程中，工程师根据不同阶段的不同设计需求选用对应的设计方法，使用范围最广泛的是周期精确级别的仿真技术，因为周期精确级别的仿真可以最精确地对电路进行仿真，并且通常仿真模型是可综合电路。然而受限于仿真速度，一个大规模的测试经常需要执行几天，因此使用硬件仿真来对周期精确级别的仿真进行加速是一个很好的方案。

# 1.1.2 仿真平台

目前主流仿真平台有软件仿真平台、硬件仿真平台和混合仿真平台，不同仿真平台具备不同的特性，可以实现不同级别的仿真技术。

软件仿真平台扩展性最强，可以实现所有精度的仿真技术，指令级别的仿真是通过软件模拟器实现，比如 QEMU 等模拟器，事务级别的仿真可以 SystemC 语言来对电路建模仿真，周期精确级别的仿真则是使用仿真工具对 RTL 语言仿真。

硬件仿真平台只适合周期精确级别的仿真，因为电路的设计必须可综合才能在硬件上实现，目前主流实现方案是FPGA原型验证（Prototype）和仿真加速（Emulation），FPGA原型验证可以加速RTL的仿真，在实际FPGA硬件上高速运行，更接近实际硬件的工作速度，同样用于电路验证以及早期应用开发，缺点是很难在FPGA原型验证板上调试设计[8]。FPGA仿真加速是指使用专用硬件来直接加速RTL仿真，可以在电路执行过程中获取到电路内部信号，虽然仿真加速的RTL仿真速度相比FPGA原型验证慢一些，但是更有利于调试设计，缺点是成本很高[9]。

混合仿真平台是结合软件仿真平台和硬件仿真平台实现的，软件仿真模型与硬件仿真模型互相结合，混合仿真平台结构如图1-1所示。

![](mineru_assets/混合仿真平台中周期精确硬件模型FPGA加速实现_郑宇辰/images/2e7b977099c70f7ab09e095ae6ad1e7dcb80931e6ce5659298143b5c274f17ce.jpg)
图1-1 混合仿真平台结构

混合仿真平台结合了软件仿真平台和硬件仿真平台的优势，不仅具备了软件仿真平台的高速、低成本特性，还可以实现RTL的高速仿真。在软件仿真平台进行指令级别或事务级别的仿真，比如对处理器等电路使用指令级别或者事务级别的仿真，在硬件仿真平台使用周期级别的仿真，对RTL设计的硬件外设，结合软硬件优势，可以大大提高电路开发效率，缩短开发周期。

软件仿真模型和硬件仿真模型之间通过Adaptor层实现数据交互[10]，软件仿真模型中对硬件进行访问时，Adaptor层将软件访问请求转换成相应的硬件访问请

求发送到硬件仿真模型中。

与纯软件仿真相比，混合仿真方法利用FPGA对Verilog硬件外设模型进行加速，以提高仿真性能，同时对于商业购买的一些IP核中只有Verilog硬件电路而没有软件模型，这样可以直接将IP挂载在硬件FPGA上，进而节省开发软件模型的时间。与纯硬件仿真相比，使用软件仿真器能够在保证高速、准确地模拟处理器核心的行为的基础上，支持多种处理器架构，并且软件仿真节省硬件资源、调试方便有效降低成本。

# 1.1.3 研究目的与意义

综合以上因素，本课题针对混合仿真平台技术进行研究，研究适用于仿真加速的硬件仿真模型，对 Verilog 周期精确硬件模型实现原理进行研究，开发工具在 FPGA 上的自动化部署以进行加速功能的实现，将这个特定的硬件仿真模型实现在混合仿真平台中，实现适合系统级大规模仿真的混合仿真平台。

硬件加速后的周期精确级别仿真模型具备高速的仿真能力，在保证周期级仿真精度的情况下实现了高速特性，具备良好调试能力的同时成本很低。集合了周期精确硬件模型的混合仿真平台能够高速进行所有级别的仿真，不但适用于大规模系统的仿真，而且还可以应用于早期应用程序开发，所以使用此混合仿真平台进行大规模电路可以大幅度缩减开发周期。

# 1.2 国内外研究现状

国内外企业和高校都对仿真技术进行了深入的研究和应用，特别是基于硬件仿真加速技术的混合仿真平台方面，接下来对国内外研究现状分析。

# 1.2.1 国外研究现状

基于硬件仿真加速技术实现的仿真工具商业应用在国外主要由各大EDA厂商进行。

三大EDA厂商均推出各有特点的硬件仿真加速工具，提供了软硬件混合仿真平台提高仿真速度。Synopsys的ZeBu中实现了专用的仿真架构，使用最先进的FPGA实现硬件加速，能够实现按多块FPGA扩展实现资源[11]；Cadence的PalladiumZ1采用的方案是使用大量处理器并行运算，实现硬件模型的并行化加速，最新产品也加入了FPGA进行加速[12]；西门子（原Mentor）的Veloce的方案类似FPGA，但是自主设计了Crystal3SoC芯片，该芯片中包含可编程的ASIC电路，专门用于仿真加速功能[13]。

除已有的商业工具外，国外学术界也有一些项目可以用于硬件加速仿真，其中典型的案例分别是 MIT（Massachusetts Institute of Technology）的 BSC（Bluespec）工具[14]、UCB（University of California, Berkeley）的 FireSim 平台和早稻田大学的 SCR（Shared Communication Register）混合仿真方法学[15]。

Bluespec 工具由 Bluespec 公司开发，Bluespec 是一个编译器工具，用于编译专用的 BCL(Bluespec Codesign Language) 语言，也叫 BSV(Bluespec System Verilog)，MIT 的计算机科学与人工智能实验室（CSAIL）基于 BSC 工具开发了软硬件接口自动化实现功能。

FireSim 平台是 UCB 开发的一个易用、可扩展的 FPGA 加速周期精确平台，通过将定制的 FPGA 加速卡接入到亚马逊云服务器中，用户可以直接通过访问云服务器，安装驱动然后使用 FPGA 仿真加速功能[16]。UCB 提供了编译器工具实现模型自动化布局到 FPGA 上，编译器工具是 Golden Gate 工具[17]，用于编译 FIRRTL（Flexible Internal Representation for RTL），FIRRTL 是一种数字电路的中间表达形式，相关工具通过 Scala 语言编写实现，能够输出 Verilog 代码[18]。

日本早稻田大学的 SCR 架构是一种 SoC 的软硬件协同验证方法，该方法基于 C/C++ 模拟器与低成本的 FPGA 仿真器的整合。模拟器和仿真器之间的通信通过基于共享通信寄存器的灵活接口进行。这种方法可以轻松进行调试，具有丰富的可移植性和高验证速度，且成本低廉。SCR 的架构如图 1-3 所示，软件仿真器与 FPGA 硬件仿真器实现了混合仿真，进行 SoC 的软硬件协同验证。

# 1.2.2 国内研究现状

国内相关EDA工具发展较慢，目前提供FPGA仿真加速方案的有国微思尔芯、亚科鸿禹等公司。

亚科鸿禹公司设计的SEMU工具能够实现指数级验证提速，工具链支持自动化流程，支持仿真大规模电路应用，支持混合仿真功能；国微思尔芯提供的“芯神瞳”协同仿真套件功能类似，同样能够实现快速FPGA原型验证和协同仿真，支持全自动化编译，进而加速验证速度和软件开发效率，大大缩短芯片项目研发周期。

国内同样也有高校进行仿真加速相关研究，清华大学、中国科学院计算技术研究所均有相关软硬件混合仿真的研究。

# 2.1 周期精确硬件模型介绍与理论分析

# 2.1.1 周期精确硬件模型介绍

目前主流的数字电路通常使用 Verilog 设计，Verilog 是一种用于设计和验证数字电路的硬件描述语言（Hardware Description Language, HDL），Verilog 语言具备对数字电路时序行为的描述能力，在 Verilog 语言中可以使用 always 块等语句来描述电路在特定时钟周期内的状态变化，周期级别的描述可以帮助工程师在电路设计过程中考虑时序约束，确保电路在给定的时钟频率下正确工作。

在一个数字芯片设计流程中，通常需要先用 Verilog 建立电路的 RTL 模型，然后进行仿真，基本的仿真测试通过后在 FPGA 中进行原型验证。Verilog 电路模型在软件仿真过程中可以很好地设计、测试和验证，但是相对的，软件仿真速度比较慢。Verilog 电路模型在 FPGA 上原型验证时执行速度较快，但是调试能力比较差，需要使用硬件调试工具比如 ILA（Integrated Logic Analyzer），难以进行周期精确级别的仿真。因此为了增强 Verilog 模型在 FPGA 硬件上的调试能力，实现仿真加速功能，就需要使用到周期精确硬件模型。

周期精确硬件模型（Cycle-Accurate Hardware Model）是一种精确描述数字电路在每个时钟周期行为的硬件仿真模型，适用于硬件上的仿真加速功能实现。课题中需要实现的周期精确硬件模型是指把一种 Verilog 电路转换成特别的电路形式，进而增强电路的调试能力，修改后的电路模型可以进行大规模测试，同时周期精确功能可以使电路进行周期基本的大规模功能验证。

常见的 Verilog 软件仿真工具具备以下几种典型功能特点:

(1) 按周期级的精度对电路进行激励以及模块引脚信号提取;
(2) 支持大规模验证程序；
(3) 可以控制电路执行, 指定硬件电路执行的周期数目, 不执行时电路状态保持。

周期精确硬件模型实现后，可以将模型综合实现成 FPGA 上的电路，修改后的模型电路在 FPGA 上执行速度比 Verilog 软件仿真器快很多，同时实现的周期硬件模型能够具备以上几种软件仿真工具的基本功能特点，因此能够实现周期精确

硬件模型在FPGA上的硬件加速，完成仿真加速功能的实现。

实现的周期精确硬件模型适合大规模电路设计的仿真加速，可以以 $\mathrm{C} / \mathrm{C}++$ 等更高级的语言来编写的测试向量，对一个大规模的IP进行充分的验证，同时模型的实现不仅仅加速了大规模集成电路IP的仿真，还可以将模型集成进入更大规模的仿真平台中。

# 2.1.2 周期精确硬件模型理论分析

周期精确硬件模型能够精确描述数字电路在每个时钟周期行为，将一个由Verilog描述的电路进行修改，转换成特别的电路形式，然后将电路实现在FPGA上实现仿真加速功能，增强在FPGA上对硬件电路的调试性。

电路形式的转换最重要的是不影响电路原有的功能，因此，为了避免影响原有电路功能，最重要的就是需要严格按照一个准确的方案进行实现，一个很合适的方案是参考MIT的CSAIL实验室提出的延迟不敏感边界数据流网络（Latency-Insensitive Bounded Dataflow Networks）电路实现，即将目标电路实现成LI-BDN形式[17,21-24]。

LI-BDN是一种针对数字系统设计和验证的特殊电路形式，旨在提高设计的可扩展性和可重用性，同时保持高效的执行性能。通过数据流网络模型对电路进行建模，并在此基础上引入延迟不敏感（Latency-Insensitive, LI）的概念，从而使得设计更容易适应不同的延迟条件。

LI-BDN 电路具备延迟不敏感特性，延迟不敏感是指一个电路或系统对于其组件之间延迟的变化具有鲁棒性。在延迟不敏感的设计中，模块之间的通信延迟可以在一定范围内发生变化，而不会影响整个系统的正确性和功能。这意味着在设计过程中，即使组件之间的延迟出现波动，也不会导致系统出现错误或失效。因此这种形式的电路可以精确控制电路的执行，并按照周期级别的精度来输入激励和提取引脚信息，符合周期精确硬件模型中调试功能的需求。

将一个普通的同步时序电路SSM（Synchronous Sequential Machines）转换成LI-BDN形式的电路通常包含三步。

第一步是将SSM转换为patientSSM。

同步时序电路即能够实现指定功能的组合电路和寄存器的组合，patient SSM是在基础的同步时序电路的基础上引入了全局控制信号，当控制信号使能时，电路中的寄存器等存储元件才能发生信号变化，具体实现过程如图2-1所示。

![](mineru_assets/混合仿真平台中周期精确硬件模型FPGA加速实现_郑宇辰/images/9aff521823348a45b9305d1a1be2e1b741c6e832822d6fec9beca7d9a820d003.jpg)
图2-1SSM转换为patientSSM示例

图2-1中展示了一个目标SSM电路转换成patient SSM电路的过程。目标SSM中有输入端口和时钟信号，时钟信号控制触发器，触发器类型包括普通触发器DFF和带有使能端的触发器DFFE。转换成patient SSM之后输入端口增加了全局使能信号target-fire，target-fire有效时，所有的触发器才会工作，否则将忽略时钟信号不寄存数据。同时为了实现控制功能，原有的普通DFF被转换成DFFE触发器，控制信号接入到target-fire上，原有DFFE触发器的控制信号与全局使能信号target-fire进行与操作。这样即可完成将一个SSM转换成patient SSM的过程。

第二步是将patient SSM实现成BDN电路。

BDN 电路是各个同步时序电路节点通过队列 Queue 互联实现，但是节点中的同步时序电路都一定是 patient SSM，因为当队列为空时电路节点并不能接入数据并发生数据传输，外部队列和 patient SSM 的组合称作原型 BDN（primitive BDN）。一个门电路的原型 BDN 实现如图 2-2 所示。

![](mineru_assets/混合仿真平台中周期精确硬件模型FPGA加速实现_郑宇辰/images/7d8447ffd31933aeae3c87ad50509ea890889b993a50a36e9ba2561f8be11bc8.jpg)
图2-2 原型BDN电路实现示例

如图2-2中可以看出，实现原型BDN电路首先需要一个已经转换成patientSSM形式的电路，然后将patientSSM形式的每一个电路端口连接一个队列，用队列来寄存每一个端口的输入输出数据。patientSSM和其他队列在一起打包在一个sim wrapper中，暴露出所有队列的端口用于与外部其他原型BDN的patientSSM

连接。

这六个端口与patient SSM的具体连接关系如图2-2中所示，电路端口的输入端口连接到输入队列的数据输出端口deq_bits，电路端口的输出端口连接到输出队列的输入数据端口enq_bits。所有输入队列的deq_valid信号和所有输出队列的enq_ready相与输出target-fire信号，表示当输入队列准备输出数据且输出队列准备好输入队列时，电路可以执行一个周期，输入队列中的每一个数据对应一个周期中目标电路的输入数据，输出队列中的每一个数据对应每一个周期中目标电路的输出数据。target-fire使能信号被接入到输入队列的deq_valid端口和输出队列的enq_valid上，表示此时输入队列应该弹出一个数据，输出队列应该压入一个数据。同时target-fire使能信号还被接入到patient SSM中，表示当前的电路可以执行一个周期。

这里可以看出将电路实现成patient SSM形式的意义，可以通过队列精确控制每一个周期的输入数据，当输入队列和输出队列没有准备好时，电路的使能信号也将无效，此时电路将忽略所有的无效输入数据并且暂停执行状态。

最后一步是将BDN电路实现成LI-BDN的形式。

一个BDN电路被称为LI-BDN电路需要在实现过程中保证两个属性，如果一个BDN电路具备这两个属性就可以被称作LI-BDN电路形式。这两个属性分别是SC（Self Cleaning）和NED（No Extranenous Dependencies）属性。

SC 属性要求：当一个模型将 N 个数据输入到其每个输出端口时，最终也必须从其每个输入端口弹出 N 个数据。

NED属性表示，若一个BDN模型的输出队列收到的数据量不超过其他任何通道，并且在模型已获取该输出通道下一个输出数据所需的全部输入数据时，该模型必须最终将相应的输出数据压入到输出队列中，且此过程与外部活动无关，全部

在一个顶层模块中完成。在这里，模型入队输出数据等同于相应的输出通道“接收”该数据。这个属性要求模型在接收到足够的输入数据后，立即生成相应的输出数据，而不受其他通道影响。

保证这两个属性可以避免BDN网络在实现过程中的一些硬件电路死锁问题，是周期精确硬件模型实现过程中的两个重要准则。

通过以上三个步骤，可以将一个普通的SSM电路转换成LI-BDN的形式，LI-BDN形式的电路有很多优势：

(1) 增强对电路的调试能力：延迟不敏感特性保证了对电路的调试能力增强；
(2) 增强电路稳定性: LI-BDN 电路实现了 SC 和 NED 属性, 保证了电路中模块与模块之间的稳定性, 避免了电路网络之间的死锁问题;
(3) 优化资源利用：可以通过将电路转换为 LI-BDN 形式，设计人员可以更好地优化资源利用，例如减少电路中的无用操作，提高电路的执行效率。这有助于降低功耗和提高集成电路的性能。此外，每一个 BDN 的电路节点都在所有输入数据准备好时继续执行，而不需要等待模块外无关的数据，可以提前执行而提前准备好输出数据，并且其他不需要执行的模块将不会工作，也会有效降低一部分动态功耗；
(4) 模块局部优化: LI-BDN 遵循延迟不敏感原则, 各电路节点可以在合适的时钟域下独立工作, 而无需同步到全局时钟, 不同的电路节点选择最佳的时钟频率, 以优化性能、功耗和资源利用。

# 2.2 硬件仿真模型连接结构

以上理论介绍了周期精确硬件模型，周期精确硬件模型可以作为混合仿真平台中的硬件仿真模型，用于对 Verilog 数字硬件电路仿真，并且在 FPGA 中的实现能够带来仿真加速效果，根据周期精确硬件模型的理论分析，将目标电路实现成 LI-BDN 的形式能够符合硬件仿真模型的需求。

根据 LI-BDN 电路形式的转换过程以及周期精确硬件模型实现的实现方案可以发现，LI-BDN 形式的电路暴露出队列的端口所以不适合直接与外部总线接口连接，而且由于上层软件仿真模型访问硬件的频率和速度都不可控，更是无法做到周期级别的控制。

所以要求实现硬件接口模块，与上层仿真模型和下层周期精确硬件模型连接，所以对应的硬件仿真模型连接结构如图2-4所示。

![](mineru_assets/混合仿真平台中周期精确硬件模型FPGA加速实现_郑宇辰/images/573e19b0aacb0a34208f949d1ddba50c63b8700c0734159f6718f7a3daa30cec.jpg)
图2-4 硬件仿真模型连接结构

从图2-4中可以看出，周期精确硬件模型即硬件仿真模型的具体实现，与Adaptor层的硬件接口共同包含在Top模块中，在Top模块中硬件接口与周期精确硬件模型通过输入输出队列连接，硬件接口与Top模块外的软件仿真模型相连进行数据交换。

根据周期精确硬件模型要求，设计硬件接口需要实现如下功能：

(1) 与 LI-BDN 电路的输入输出队列连接, 硬件接口自动化压入和弹出数据;
(2) 对电路模型实现周期级别精度的控制,可以随时、重复读取当前时钟周期下的输入输出信息;
(3) 复位功能实现，能够对所有的队列复位。

将一个电路转换成 LI-BDN 电路的形式比较复杂，需要修改所有的内部寄存器模块，并且设计模块连接的队列，以及实现模块之间的互相连接，最后还需要与对应的硬件接口连接，因此对于一个大规模的电路进行手动 LI-BDN 化是比较困难的。但是从步骤转换中可以看出，每一步的转换都存在一定的规律性，使用编译器进行自动化流程实现是比较合适的方案，所以课题中修改了编译器工具，以实现

周期精确硬件模型生成的自动化。

一个比较合适的实现方案是基于开源 Yosys 编译器工具进行修改，接下来介绍 Yosys 编译器工具。

# 2.3 Yosys编译器工具与RTLIL分析

Yosys 是一个基于 C++ 语言开发的、开源的 Verilog HDL 综合工具，由 Clifford Wolf 开发，旨在为开源硬件设计提供一个高质量、灵活且可扩展的综合工具[25,26]。它在数字电路设计中发挥着重要作用，将硬件描述语言表示的设计转换为电路元件（例如，查找表、寄存器等）。

Yosys 通过脚本化的命令语言提供高度可扩展性，用户可以轻松地扩展和自定义综合流程，以满足特定需求。此外，它具有高度模块化的设计，允许用户轻松地为其添加新的综合步骤和优化算法。这种模块化的设计不仅便于维护和升级，而且有助于提高工具的可靠性和性能。

Yosys 支持多种输入和输出格式，如 Verilog、BLIF、EDIF 等，可以轻松地与其他工具链集成。此外，Yosys 还提供了丰富的命令行选项，使得用户可以根据需要灵活地配置综合流程和参数。在实际应用中，用户可以将 Verilog 源代码输入到 Yosys，然后生成 FPGA 或 ASIC 所需的网表，如 Xilinx Vivado 或 Synopsys Design Compiler 可接受的网表格式。这有助于将设计实现在硬件上，实现从前端设计到后端实现的无缝连接。

Yosys广泛应用于各种集成电路设计领域，包括但不限于开源硬件设计、教育与研究、验证与调试等。借助Yosys，设计者可以将Verilog代码转换为FPGA或ASIC所需的网表，从而将设计实现在硬件上。其开源特性使得开源硬件社区能够充分利用该工具，降低设计成本并推动开源硬件发展。

Yosys是完全开源的，让用户能够查看、修改并贡献代码，以满足自己的需求和改进工具，因此很适合本课题中对于编译器修改的需求。Yosys支持将处理后的电路生成Verilog代码，所以生成的周期精确硬件模型同样可以在不同工具中实现。

Yosys作为一种编译器，同样由前端（Frontend）、中间表达形式（Intermediate Representation，IR）和后端（Backend）组成。

Yosys编译器结构如图2-5所示。

从图2-5中可以看出，Yosys编译器包含三个关键部分，前端、中间表达形式和后端。Yosys编译器的前端主要处理硬件描述语言的语法和语义分析，解析Verilog等类型的源代码形成抽象语法树（Abstract Syntax Tree，AST)，将结果传输到AST前端转换成Yosys内部的中间表示形式RTLIL（Register Transfer Level Intermediate Language)。一些电路的优化在IR中以Pass的形式实现，Yosys编译器将直接修改RLTIL中间表达形式。Yosys编译器的后端负责将RTLIL表示的硬件电路转换成特定FPGA或ASIC平台的网表（例如EDIF、BLIF等格式）。

根据Yosys编译器结构和周期精确硬件模型理论分析，直接对Yosys编译器的中间表达形式RTLIL处理是一个合适的方案。

RTLIL 是一种针对寄存器传输级（RTL）设计的中间表示形式[25,27]，用于在 Yosys 编译器内部表示硬件电路结构。RTLIL 作为一个灵活的中间表示（IR），使 Yosys 能够在设计流程的不同阶段方便地处理和操作硬件设计。RTLIL 在 Yosys 的编译流程中起到承上启下的作用，连接着前端的硬件描述语言（如 Verilog、VHDL 等）解析和后端的综合优化、目标平台实现等功能。作为中间表示，RTLIL 不仅表达了源代码中的硬件结构信息，还支持编译器进行各种优化和转换操作。

从图2-6中可以看出，数据结构中包含Design、Module、Cell等多种类型，不同类型分别对应RTL级或者门级的一些硬件电路描述。其中Design指整体的RTL设计，在一个电路的处理过程中只有一个Design，Design中包含若干个Module，每一个包含的Module都对应Verilog语言中对一个模块的例化。

一个Module中包含若干个Cell、Wire、Process和Memory数据结构，Cell表示基本逻辑单元或子模块比如常见的逻辑门、触发器，Wire表示信号线，用于连接Cell之间的端口。Process表示过程块用于描述组合逻辑和时序逻辑。Memory表示内存单元，用于存储和访问数据。这些数据结构共同构成了模块内部的电路结构和行为，实现了电路设计的表达和分析。Module和Cell的关系是：Module中包含Cell，一个Module被实例化后，对应的Module被实现成一个Cell。

在一个过程块的描述中还涉及一系列特殊规则，由于RTLIL中的操作通常通过控制流结构来表示，如顺序块、条件语句和循环。顺序块表示一系列按顺序执行的操作，条件语句允许基于信号值选择不同的操作路径，循环则允许重复执行某些操作。这一系列特殊规则就是为了表示这些规则，特殊规则包括SyncRule、CaseRule和SwitchRule。SyncRule用于表示在特定时钟边沿上同步的操作，CaseRule表示基于信号值的多路径选择，SwitchRule表示开关选择逻辑。这些规则使RTLIL能够准确地表示硬件设计中的控制和数据流行为。

除图2-6包含的数据结构之外，还有一部分关键结构，一个关键的数据结构是用于描述模块之间连线规则的SigSpec和SigSig数据结构。SigSpec和SigSig数据结构用于处理和操作信号信息，它们是实现电路表示和转换所需的关键组件。SigSpec是一个表示一组信号的数据结构。它由一系列SigBit组成，用于描述一组有序的位。在RTLIL中，信号可以是简单的布尔值或位向量，它们用于表示数据路径中的值。SigSpec可以表示一个或多个位，例如多位总线或向量信号。SigSpec通常用于定义信号之间的连接、数据路径或操作数。SigSpec数据结构提供了许多

用于操作和查询信号的方法，例如获取信号的宽度、拆分或合并信号、以及访问特定位等，SigSpec 的创建通常可以由 Wire 类型实现。SigSig 是一个表示信号对的数据结构，通常用于描述信号之间的连接。SigSig 包含两个 SigSpec 对象，一个表示连接的源信号，另一个表示连接的目标信号。SigSig 数据结构用于在硬件设计中定义信号之间的连接关系，例如在模块间的连线、内部数据路径或逻辑单元之间的连线。SigSig 数据结构可以通过两个 Wire 类型或者两个 SigSpec 类型的数据创建，表示将这两个数据相连。

从 LI-BDN 电路形式分析过程中可以看出，在一个电路 LI-BDN 化的过程中，对于原有电路，只涉及电路的寄存器以及端口的改变，对于这些改变，可以在门级直接进行实现，而不需要在 RTL 级别处理。因此为简化 Yosys 开发流程，可以选择将输入的 Verilog 电路设计先转换成门级，Yosys 中提供了 proc 等命令，用于将一个电路综合成门级。当电路实现成门级时，电路中没有了过程块的描述，所以在 RTLIL 的 Module 中，只需要关注 Cell、Wire 和 Memory 类型。

从 LI-BDN 电路形式分析过程中可以看出，需要直接对电路形式进行修改，因此，根据 Yosys 结构，直接修改 Yosys 中电路的 RTLIL 来实现周期精确硬件模型是一个比较合理的方案，接下来介绍 Yosys 编译器源码的修改，即周期精确硬件模型在 Yosys 编译器中的实现。

# 2.4 实现周期精确硬件模型

Yosys 实现周期精确硬件模型的依据一个普通电路 LI-BDN 化的过程，即具体实现就是根据 LI-BDN 化流程修改 Yosys 相应的源码，对 RTLIL 中间表达形式处理，直接生成相应电路。

根据第二章中对 LI-BDN 电路形式具体分析，总共包含三步，第一步是将 SSM 转换为 patient SSM；第二步是将 patient SSM 实现成 BDN 电路；最后一步是将 BDN 电路实现成 LI-BDN 的形式。其中在 Yosys 实现中，我们在将第二步实现中直接保证 SC 和 NED 属性，这样可以节省第三步的步骤。所以电路 LI-BDN 化在 Yosys 实现中总共需要两步，一步是将 SSM 电路转换成 patient SSM 形式，另一步是将 patient SSM 电路 LI-BDN 化，并生成延迟不敏感的接口，流程图如图 2-7 所示。

第一步是为每一个模块增加全局使能信号 target-fire 的端口，用于全局信号的接入，这部分操作的是 RTLIL 数据结构中的 Module 类。端口新增具体体现成使用 addWire 函数，创建 Wire 信号，并配置宽度为一，属性为输入，表示信号由外部输入。所有 Module 都需要增加这一端口，所以遍历 Design 中的所有 Module 进行此操作。

第二步是将模块的全局使能信号 target-fire 连接，具体体现是一个 Module 中实例化出一个 Cell，然后这个 Module 将内部的全局使能信号与 Cell 的使能信号输入端连接。这一步骤循环进行，遍历 Design 中的所有 Module，但是不是所有 Module 都需要进行连接，因为有 Module 中没有实例化非标准库的 Cell，判断 Module 为此类型时不需要进行此步骤可以直接跳过。

第三步是将全局使能信号与触发器实现连接，这一步中检测了RTLIL中所有的Cell模块，如果检查出所有的DFF和DFFE的触发器，然后将这两部分的触发器进行信号重新连接。以DFF触发器的修改为例，提取DFF原有的输入端（D）、时钟端（CLK）和输出端（Q），然后使用addDffe创建新的DFFE模块，将原有的三个端口连接到新的DFFE触发器对应端口上，最后将全局使能信号连接到DFFE的使能端（EN）上。同样，这一步也需要遍历所有Module进行。

完成这三部后可以实现 Yosys 编译器中将 SSM 电路转换成 patient SSM 的功能，具体体现为一个电路处理的 Pass，当编译器需要进行此变换时，只需要调用相应的命令即可，注意需要在调用前先调用 proc 将 RTL 级电路综合成门级。

相应 patient SSM，并且将 patient SSM 的端口与外部队列相连，共同包含在一个 simwrapper 中。

这一步骤中不仅需要考虑端口与外部队列的相连，还要进行队列模块的设计，整体流程具体流程如图2-9所示。

首先需要创建一个顶层Module，名称是simwrapper，然后实例化顶层目标模块的Module，实现成simwrapper中的一个Cell。

之后需要进行端口创建和信号提取，首先为sim wrapper创建端口，固定的需要创建的端口有时钟端口clock和复位端口reset，需要注意的是这里的复位端口reset不是用于电路内部寄存器的复位，而是所有队列的复位。然后提取出顶层目标模块中的时钟信号，将时钟信号与时钟端口clock连接。

接下来需要设计队列硬件模块，队列模块在方案中根据端口数目进行一对一的实例化，且在RTLIL的级别不需要关注队列模块内部的信号，所以将队列模块直接用Verilog语言描述是一个合适的选择。队列模块的主要功能是入队和出队操作。当新数据到达时，可以将其存储到队列的下一个可用位置。当数据需要被处理时，可以从队列的头部提取。这种数据处理方式确保了数据按照到达顺序进行处理，实现了先进先出（FIFO）的特性。并且考虑到端口的宽度不固定，以及队列的深度对性能可能的影响，所以队列设计中充分考虑了参数化，使用parameter定义了DATA_WIDTH和QUEUE_DEPTH表示队列中的数据位宽和队列深度。最后在实现了这个Verilog的队列模块后，在Yosys源码中调用read_verilog命令读取队列模块，之后可以根据队列模块的名称进行实例化，并能够配置其中parameter定义的参数。

在实现了队列模块后，需要根据每一个有效数据端口进行相应的信号连接，其中原有目标电路的复位信号也看作为一个普通的数据端口，也就是说对于一个patient SSM的端口，除去时钟端clock和全局使能信号target_fire的所有剩余端口都需要与一个队列连接。队列连接时遍历patient SSM剩余端口，需要根据原有端口的不同的输入输出方向有相应的连接方式。对于输入信号，需要实例一个队列，

将队列的输出信号连接到 patient SSM 的输入信号上，同理对于输出信号，需要实例一个队列，将队列的输入信号连接到 patient SSM 的输出信号上。

输入信号对应队列的 enq_valid、enq_bits 和 enq_ready 作为 sim wrapper 的端口接出，同理输出信号对应队列的 deq_ready、deq_bits 和 deq_valid 也作为 sim wrapper 的端口接出。接出信号时，保证每个信号都使用原有信号的端口，作为信号名称的后缀，方便后续信号连接时的信号匹配。

最后一步是实现内部全局使能信号 target-fire 的产生和连接。根据 LI-BDN 电路形式分析，可知是所有输入队列的 seq_valid 信号和所有输出队列的 enq_ready 相与输出的 target-fire 信号。在对 RTLIL 实现的具体操作是，首先创建 Wire 类型变量 target-fire，然后提取出相应的所有信号，这些信号逐级两两相与，与门的创建是调用函数 addCell 并指定 Cell 的类型是与门，然后调用 connect 函数将最后一级与门的输出与 target-fire 相连。然后是将 target-fire 连接到所有输入队列的 seq_ready 和输出队列的 enq_valid，以及 patient SSM 的使能端口，这一步骤具体是调用相应 Cell 的 setPort 函数，函数指定需要连接的端口名称并传入根据 target-fire 创建的 SigSpec 类。

以上步骤完成后可以实现这一个将patient SSM进行LI-BDN化的Pass，此电路LI-BDN化过程在满足周期精确硬件模型的需求情况下，进行了一些简化，主要体现在没有对硬件电路进行具体划分，只在顶层模块使用队列互联，因此电路一定保证了SC和NED属性，不会出现死锁等问题。

在电路处理之外需要实现的是根据patient SSM的端口信息生成头文件，用于后续进行硬件接口设计以及软件仿真模型实现。课题中根据后续需求，实现了两个头文件的自动化生成，一个的用于Verilog调用的头文件，一个是用于C/C++语言软件仿真模型调用的头文件。这两个文件在这个Pass的内部实现，具体是在电路生成之后，在两个头文件中保存了所有端口的名称，并为所有端口依次分配了相应的ID号，用于后续的API调用。同时保证两个文件中的ID一一对应，便于后续软件仿真模型对硬件接口访问。

以上就是电路 LI-BDN 化生成周期精确硬件模型的所有具体实现，但是正如第二章中的介绍，不能直接将这个电路进行应用，还需要设计相应的硬件接口，接下来介绍硬件接口的具体实现。

# 2.5 硬件接口实现

根据对硬件接口的功能要求，进行相应的实现。主要包括与周期精确硬件模型

实现控制功能和实现软件通信接口两部分。

# 2.5.1 周期精确硬件模型控制功能实现

需要实现的控制功能包括以下几种：

(1) 与电路的输入输出队列连接，硬件接口自动化压入和弹出数据；
(2) 能够对电路模型实现周期级别精度的仿真控制;
(3) 实现复位功能。

根据对周期精确硬件模型方案发现，硬件接口需要与周期精确硬件模型中各个队列的不同端口进行连接，因端口数量和方向不固定，所以修改Yosys编译器，对电路的RTLIL进行修改是一个比较合适的方案，实现流程如图2-10所示。

![](mineru_assets/混合仿真平台中周期精确硬件模型FPGA加速实现_郑宇辰/images/94278361ec811126b2fa0bdad67b59cc547124543421a9af2c7f90c88f5fe59b.jpg)
图2-10 硬件接口模型连接与控制功能实现流程

首先要实现与电路模型输入输出队列连接，并且控制自动化压入和弹出数据，需要在硬件接口中设计一个信号 execute，当 execute 信号为高时标志软件仿真模型发出命令控制电路模型开始执行，并且电路执行还需要此时电路模型中所有输入队列的 enq_ready 信号和所有输出队列的 deq_valid。当电路开始执行时，所有的输入队列的 enq_valid 和输出队列的 deq_valid 会被拉高，此时输入队列开始压入数据，在输入队列压入数据后，LI-BDN 内部的全局使能信号有效，patient SSM 开始执行，并且输出队列会弹出数据。这样就实现了硬件接口对周期精确硬件模型的执行控制，进行输入输出队列的自动化压入和弹出数据。

之后要实现电路的仿真控制功能，保证电路模型的周期级别精度的控制，这里在硬件接口中设计sim_step信号，sim_step每个周期会减一，当sim_step不为零时execute信号有效，并且电路会持续执行，上层软件仿真模型会发出控制指令重新写入sim_step的值，sim_step信号保证了电路模型执行的周期数目。同时为了保证精准控制，还需要实现sim_done信号，用于表示当前电路模型已经执行完成，需要注意的是不能仅根据sim_done的值是否判断，还需要考虑所有输出队列的数据是否全部弹出以及是否正在压入数据，并且硬件接口不能处于复位状态。当sim_done信号拉高时，表示电路已经执行完成，在软件仿真模型读取硬件接口的状态信息需要将此数据返回。

最后需要实现的功能是电路模型复位，这里的复位不是指对电路内部的寄存器复位，而是复位电路模型中LI-BDN化后生成的所有队列，把所有队列清空，需要实现sim_reset信号来控制对电路模型中队列的复位，同时当硬件接口被复位的

时候电路模型中的队列同样会被复位。考虑到对软件仿真模型的操作简化，sim_reset信号有效一个周期后就会被自动清空，可以节省软件再次控制sim_reset清空的一次数据传输。

# 2.6 顶层模块实现

完成了周期精确硬件模型的自动化生成和硬件接口的自动化生成后，需要将这两个关键模块进行互联，自动化生成顶层Top模块，并在FPGA上实现。

这顶层模块的自动化实现中，修改Yosys源码首先需要调用read_verilog命令读取Verilog文件，以硬件接口模块的接口部分为基础进行RTLIL的修改，在这个Module中创建新的端口与队列连接，需要注意的是原有的sim wrapper的输出端口对应着硬件接口模块的输入端口，反之亦然。

原有设计的每个端口都对应队列的三个端口，分别是bits端口、valid端口和ready端口，为了实现对硬件模型的控制，首先需要根据电路模型原理分析，从LI-BDN电路形式可以看出，LI-BDN化的电路是数据流驱动，每接收到一个数据，就会处理一个数据，没有数据到来时将一直等待。

通过表2-2中的几个信号功能介绍中看出输入输出队列的相应valid端口和ready端口的连接方式，可以看出数据流的控制对应到输入队列中就是对输入队列的enq_all_valid信号进行控制，而在硬件接口中的体现就是插入与门，将from_host_ready和execute信号相与，通过execute信号控制数据流。当execute信号有效时，硬件接口向输入端口的队列压入数据，进而控制电路执行一个周期。当执行完成后会把数据压入到输出队列中，然后将数据寄存在相应的regs寄存器中。

完成所有队列的 valid 和 ready 端口连接后，最后是 bits 端口的连接，这部分

是将输入输出队列的端口与上文中的中间变量 bits_input 和 bits_output 相连，具体实现是调用 connect 函数，同时在声明 SigSpec 类时实现了端口位宽的参数化配置。

最后一步是将硬件接口和周期精确硬件模型共同例化在一个顶层 Top 模块中，并且为 Top 模块设计输入输出端口，这样就完成了顶层 Top 模块的设计。

# 2.7 本章小结

本章解析了周期精确硬件模型及其在仿真加速实现中的作用，以及如何将同步时序电路转换为 LI-BDN 电路，这一电路具有良好的可扩展性、可重用性以及可调试性。分析了硬件仿真模型的连接结构，强调了在 FPGA 硬件实现中软件仿真模型的重要性。

# 第3章 混合仿真平台实现

根据混合仿真平台结构，实现了硬件仿真模型以及硬件接口后，需要对混合仿真平台的剩余部分进行实现，包括两个混合仿真平台方案，分别是基于Zynq的嵌入式混合仿真平台和基于QEMU的混合仿真平台。

# 3.1 基于Zynq的嵌入式混合仿真平台实现

这部分混合仿真平台基于已有的硬件仿真模型和硬件接口，需要实现软件仿真模型以及Adaptor层中剩余的部分，即软件接口和软硬件通信，其中软件仿真模型与具体硬件功能相关。此方案设计中，嵌入式混合仿真平台的所有部分都实现在Zynq中，首先介绍此混合仿真平台实现方案。

# 3.1.1 实现方案

一种典型的混合仿真平台实现方案是基于Zynq实现的嵌入式混合仿真平台，Zynq是赛灵思（Xilinx）推出的一款具有高度集成特点的可编程SoC（System on Chip）平台。ZYNQ平台主要包括两个核心部分：可编程逻辑（Programmable Logic, PL）和处理器系统（Processing System, PS）[28]。

可编程逻辑部分主要由FPGA组成，可以实现高度定制化的硬件电路设计。PL中包含了查找表（LUTs）、触发器（FFs）、片上存储（BRAMs）、DSP切片（DSP Slices）等可配置逻辑资源，以及相应的互连结构。用户可以通过硬件描述语言（如Verilog或VHDL）来设计数字逻辑电路，并将其映射到PL上。PL部分通常用于实现对于性能敏感的任务，如数字信号处理、图像处理、硬件加速器等。

处理器系统部分基于 ARM 结构的处理器，通常包括双核 ARM Cortex-A9、ARM Cortex-A53 等处理器。处理器系统中还集成了一系列外设接口和通信模块，如 DDR 控制器、GPIO、UART、以太网控制器等。此外，PS 还包括一个内存管理单元（MMU）以支持虚拟内存地址映射，以及一个中断控制器，用于处理来自外设和 PL 的中断请求。处理器系统（PS）负责处理高层次的运算任务，如操作系统、应用程序等。

PS 与 PL 之间通过一组高速总线相互连接，如 AXI（Advanced eXtensible Interface）总线[29-31]。这种连接方式使得 PS 和 PL 可以实现高效、低延迟的数据交换，进而提高整个 SoC 系统的性能。因此 Zynq 中软硬件交互的开销非常低，比较适合课题中软硬件接口的实现。

硬件仿真模型的电路综合实现在 Zynq 的 PL 中，而软件仿真模型实现在 Zynq

的 PS 中，软件仿真模型中通过相应的软件对硬件仿真模型进行信号激励。根据 Zynq 平台的软硬件交互特性，实现的 Adaptor 层包括与软件仿真模型交互的软件接口和与硬件仿真模型交互的硬件接口，以及软件硬件互相实现数据交换的 AXI 总线。

在 Zynq 平台上实现的混合仿真平台结构如图 3-1 所示。

![](mineru_assets/混合仿真平台中周期精确硬件模型FPGA加速实现_郑宇辰/images/bf6c796282db60b1979a7c830a794639859671b63f9df30d9232d341d790c990.jpg)
图3-1 嵌入式Zynq平台混合仿真结构

从图3-1看出，软件激励执行在PS中，向Adaptor层中的软件接口发送控制信息实现，Adaptor层中软件接口将数据通过AXI总线转发到硬件接口中，然后硬件接口将对应的数据转发到周期精确硬件模型中，进而实现对电路模型的控制。

# 3.1.2 软硬件通信实现

基于Zynq平台实现软硬件通信包括三部分，分别是硬件电路实现、软件处理器核实现和软硬件通信。这几部分主要使用Xilinx Vivado工具中提供的Block Design工具设计并实现[32,33]。

课题实现中使用的FPGA开发板是ZedBoard开发板，ZedBoard是基于XilinxZynq-7000系列SoC（SystemonChip）的开发板，开发板不仅片内资源丰富，还提供了多种外部接口，符合课题需求[34]。

具体的 Block Design 设计后的结果如图 3-2 所示。

![](mineru_assets/混合仿真平台中周期精确硬件模型FPGA加速实现_郑宇辰/images/ffde618aada2411c30a46c7b41739240f431fd4e88a932df5fc0c1371ff72f8b.jpg)
图3-2Block Design设计结构图

图3-2中，Block Design包含三个核心IP核，并将相应接口接出。

图中的Processing System IP核是Zynq-7000平台中的核心IP，集成了ARM Cortex-A9的处理器。软件接口在IP中的Cortex-A9处理器核中运行，当软件访问指定外设地址时，处理器会使用总线访问外设，因此，需要将周期精确硬件模型挂载在这个处理器的总线上，并分配相应的地址。由于访问由软件接口发起，所以处理器侧作为Master端，通过AXI总线进行访问，可以看到处理器IP核有AXI总线接口，连接到AXI Interconnect IP核上。

AXI Interconnect IP 核是 Xilinx 提供的一种高性能、低延迟的总线连接模块。它主要用于在 Xilinx FPGA 或 SoC 设计中连接多个 AXI 主设备和 AXI 从设备，支持 AXI4、AXI4-Lite 和 AXI4-Stream 协议。在设计中目前只接入了一个目标电路，所以相应的只有一个从接口。

最后一个核心 IP 核是一个 AXI 接口 IP 核，作为接口将 AXI 接口与周期精确硬件模型的顶层 Top 模块相连，使用这个 IP 可以避免自己设计 AXI 总线接口，一定程度上节省开发时间，同时使用成熟 IP 核稳定性更高。

因此可以看出，软件接口在PS的处理器核中，周期精确硬件模型实现成外设挂载在AXI总线上，软硬件互相通过AXI总线通信，AXI总线互联互通核心AXIInterconnectIP核实现。使用BlockDesign来调用IP实现的优势是扩展性强，由于AXIInterconnectIP核可以自定义配置从机数目，所以可以很方便地进行外设扩展，如果增加了一个新的硬件仿真模型，可以直接挂载在总线上，而仅仅需要配置IP核参数以及例化新的AXI接口IP核。

此外，AXI Interconnect IP 核具备跨时钟域的支持，因此在这个软硬件设计中，PS 的处理器核与外设硬件电路模型是工作在不同时钟域下的。由于仿真加速的目标电路设计不同，相应的硬件仿真模型可以工作的最大时钟频率也不同，因此可以根据电路设计相应地调整时钟频率，而不影响处理器核始终以很高的频率执行，最大化保证了混合仿真平台的仿真性能。

# 3.1.3 软件接口实现

软硬件通信的数据通路实现后需要实现软件接口，软件接口通过 C/C++ 语言

实现，执行在PS的处理器核上，同时在PS的处理器核上启动Linux操作系统，作为 $\mathrm{C / C + + }$ 软件执行环境。

根据周期精确硬件模型的控制方案以及软硬件数据通路的实现，目前实现的软件接口具备以下功能：

(1) 访问指定外设地址实现对硬件仿真模型的访问;
(2) 封装访问的 API 函数，提高可移植性并且便于编写软件激励程序；
(3) 设计软件激励中与硬件接口的数据交互方案。

首先是访问指定地址的功能实现，由于软件接口执行在 Linux 操作系统的环境下，不能直接访问特定的物理地址，所以使用 mmap 函数进行地址映射，来获取对应的虚拟地址，然后再通过对这个虚拟地址的访问实现对外设的访问，然后软件接口根据这个地址进行不同的偏移就可以访问不同的寄存器。

第二步是封装访问周期精确硬件模型的API函数，对不同地址进行不同的封装，具体需要实现的地址偏移如表3-1所示，可以看出除去必要的数据端口外，还需要实现用于复位的sim_reset信号、用于控制电路执行的sim_step信号和用于检测是否执行完成的sim_done信号。根据电路仿真需求以及已有的寄存器表，可以设计软件仿真接口，实现了几种基础仿真功能函数封装。

调用表3-1中的API可以实现对硬件仿真模型的控制，进而实现混合仿真平台中的软件激励。

根据 API 功能可以设计使用软件激励进行仿真的流程，软件激励的实现中，首先调用 target_reset 函数，对整个模块的队列进行复位清空，然后开始进行每个周期的测试循环，周期内先调用 poke 函数指定熟读端口要写入的数据，然后调用 step 函数，控制周期精确模型执行指定的周期数，之后调用 peek 或 expect 函数读

取输出端口上的数据。在执行指定循环数后，要结束测试时调用finish函数，来结束软件激励程序，同时混合仿真也将停止。使用封装的API实现软件激励如图3-3所示。

# 3.2 基于 QEMU 的混合仿真平台实现

基于 QEMU 的混合仿真平台与基于 Zynq 的混合仿真平台不同，基于 QEMU 的混合仿真平台更适合进行系统级别的仿真，需要充分利用周期精确硬件模型的特性保证了仿真的精度和速度。

# 3.2.1 实现方案

基于 QEMU 的混合仿真平台是在基于 Zynq 平台的嵌入式混合仿真平台的基础上，增加了 QEMU 软件模拟器部分，QEMU 是一款开源的处理器模拟器，具备广泛应用和高性能的特点。支持多种处理器架构，如 x86、ARM、RISC-V 等[35,36]。

QEMU采用了动态二进制翻译技术，实现对CPU高效的模拟。相对于其他软件模拟器而言，QEMU在模拟速度方面具有明显的优势。同时，QEMU支持完整

的指令集和处理器特性，如异常处理、中断控制和内存管理等，为软件开发者提供了一个准确的目标处理器环境，有助于发现和解决潜在的软件问题。

根据混合仿真平台结构，将QEMU的软件激励作为软件仿真模型，设计混合仿真平台结构如图3-4所示。

从图3-4中可以看出，在原有图3-1的Zynq平台结构的基础上，基于QEMU的混合仿真平台的软件仿真模型是在QEMU的虚拟CPU中执行的软件激励，硬件仿真模型仍是周期精确硬件模型，但Adaptor层的实现更加复杂。在原有Zynq平台的基础上，在PS中设计了通信模块与QEMU中的通信模块进行跨平台数据交互，实现FPGA硬件与PC机上QEMU的交互，PS上的通信模块与软件接口进行数据交互，混合仿真平台在PS侧增加了通信模块的设计，用于与QEMU通信，PS中的通信模块将从QEMU接收到的数据与软件接口进行交互。

在上述的混合仿真平台中，软件模拟器QEMU逐级访问周期精确硬件模型完整的流程图如图3-5所示。

根据方案设计，周期精确硬件模型在混合仿真平台中集成主要有三个主要部分：

(1) 集成具备通信模块的软件模拟器作为 CPU 和总线的仿真工具;
(2) 在 Zynq 上实现通信模块与软件模拟器的通信模块进行数据交互；
(3) 在通信模块与软件接口数据交互时实现信号转换。

其中 QEMU 仿真器中的虚拟 CPU 执行软件仿真模型，QEMU 中的通信模块、Zynq 上的通信模块均属于 Adaptor 层。

# 3.2.2 QEMU 集成实现

课题中选择使用 QEMU 作为软件模拟器，具备多种优势，其中一个重要优势是 QEMU 作为开源软件，社区资源丰富，其中 Xilinx 官方提供的 QEMU 版本用于仿真 Xilinx FPGA 和 SoC 设计，相比主分支的 QEMU 工具提供了远程通信接口并且支持 DTB（Device Tree Blob）文件创建远程接口设备[37]。

Xilinx QEMU 中提供了专门用于混合仿真的通信模块，并提供了相应的通信协议。Remote Port（RP）协议就是 Xilinx QEMU 中提供的一种与仿真外部系统通信的机制。它允许 QEMU 实例与外部仿真环境或其他 QEMU 实例进行通信，以实现硬件、软件和外部设备之间的协同仿真。

Remote Port 协议的主要功能是实现 QEMU 与外部仿真系统之间的同步和数据传输，专门用于点对点传输。通过这个协议，QEMU 可以访问外部设备的寄存器、内存空间和 I/O 资源。这种机制使得在 QEMU 中运行的软件可以与实际硬件设备进行交互，从而实现对整个系统的快速原型验证。

Remote Port 协议是基于 TCP 的客户端-服务器模型实现的。在这个模型中，QEMU 充当服务器，而外部系统充当客户端。它们之间通过 TCP 连接进行通信，传输控制命令和数据，在 Linux 环境下，TCP 通信基于 Socket 接口完成[38,39]，所以软件模拟器侧的结构如图 3-6 所示。

# 3.2.4 通信模块与软件接口信号转换实现

通信模块中与软件接口需要进行信号转换，由于QEMU中通信模块的访问是事务级的，所以需要实现信号转换，完成事务级模型（Transaction-Level Modeling，TLM）信号与低层次的周期引脚精确（Pin Cycle Accurate,PCA）模型相互转换，确保在这些不同抽象层次之间进行数据转换、通信和仿真时间协调。

目前多数IP核在使用时是以总线接口的方式连接，而在系统级别的仿真中，上层QEMU软件模拟器通过通信模块传输的数据也是事务级别的，所以为了将带有总线接口的IP核快速集成计入到混合仿真系统中，就不仅需要信号转换模块来将TLM级别的信号转换成PCA级别，还需要实现TLM信号到不同总线协议的转换。

以 APB 3.0 总线[40]的一次写访问为例，Adaptor 层将一个 Transcation 转换成相应周期的操作如图 3-9 所示。

从图3-9中可以看出，一个Transaction传入到信号转换模块中，信号转换模块将这个Transcation转换成对应总线的不同周期的不同信号，图中APB3.0的写操作被转换成地址信号等多个数据，并且等待PREADY信号的反馈，等待到反馈

后结束这一次 Transaction 的传输。

图3-9 信号转换过程

信号转换模块将周期级别信号的输出软件接口中实现，基于软件接口中封装的API控制硬件仿真模型。这样实现后可以对周期精确硬件模型进行封装，系统仿真中模拟的CPU只需要关注硬件的寄存器地址以及偏移，而不需要关注硬件的总线实现方式，实现高层次封装大大增加扩展性。

在信号转换模块实现了 Transcation 与 PCA 级别信号的转换后，需要实现 Adaptor 层与通信模块的数据交互，由于两部分都是执行在 PS 侧的软件，所以需要设计相应的信号传递方案。

因为PS侧软件执行在Linux环境下，所以一个合适的方案是使用多线程的方式设计通信模块与Adaptor层的并行执行以及执行期间的数据交互。多线程中不同线程的数据读写时需要互斥锁来保证数据，同时为了避免死锁和资源浪费使用了条件变量。

# 3.3 本章小结

本章介绍了两个使用了周期精确硬件模型 FPGA 仿真加速的混合仿真平台具体实现，分别是基于 Zynq 的混合仿真平台和基于 QEMU 的混合仿真平台。

基于Zynq的嵌入式混合仿真平台充分利用了Zynq平台特性，实现了适用于周期精确硬件模型的软硬件通信方案，软硬件通信方案使用Zynq平台中的AXI总线实现，使用Vivado工具中的Block Design工具完成软硬件平台设计。软件接口基于硬件接口实现设计，实现了多种函数API封装以及软件激励方案，从而有效地实现了仿真加速功能。

基于QEMU的混合仿真平台实现了Xilinx QEMU在混合仿真平台的集成，使用DTB创建了远程通信模块。在Zynq平台的PS侧中实现了软件通信模块，并且实现了相应Remote Port通信协议，可以与Host主机的QEMU进行跨平台通信。在软件通信模块与软件接口间实现了信号转换，将TLM级别信号与PCA级别信号互相转换，并且提供了特定总线协议转换的功能。

课题中选择使用开源 FFT IP 核电路来进行功能验证，对 IP 核的 Verilog 代码进行编译，生成指定电路模型后，首先检验电路模型中是否成功插入全局使能信号 target-fire，然后需要检查 LI-BDN 化的实现，即输入输出队列的插入与连接，最后检查硬件接口生成与连接。

测试中选取具备 APB 3.0 总线接口的 SRAM 作为外设 IP 核，使用经过测试的编译器编译 IP 核并实现在 FPGA 硬件上。

由于仿真平台中使用 QEMU 作为软件仿真器，所以对应的需要将测试程序实现成在 QEMU 执行的裸机程序，并且在此基础上实现对 IP 核的访问，如果访问的结果符合电路功能，则说明混合仿真平台功能正常，周期精确硬件模型成功集成在平台中。

首先需要设计软件激励测试程序，测试程序是C语言裸机程序，并且编译为ARM架构，对于SRAM电路需要实现随机读写功能，设计裸机测试程序伪代码如图4-13所示，图中裸机测试程序伪代码实现了对SRAM的基本测试，首先定义了外部生成的随机数数组，随机数为32位无符号整型，然后定义了三个读写API函数。主函数中首先初始化UART串口用于向外打印数据，然后开始循环分别向不同地址写入不同的随机数据，并且在写入后进行读取对比，将每一次对比结果打印输出，可以查看结果中每一次输出来判断测试是否通过。

# 4.4 本章小结

本章中对周期精确硬件模型仿真加速和混合仿真平台进行了三个部分的测试。首先对周期精确硬件模型的自动化生成实现进行测试，选用开源 FFT IP 核电

路，使用修改后的 Yosys 编译器进行自动化编译，从编译生成的 Verilog 电路中逐级验证，验证了全局使能信号加入、LI-BDN 化、硬件接口生成等关键部分，使用 Verilator 对硬件电路进行模拟，观测周期精确硬件模型的波形，从波形的关键部分确定编译器工具生成的电路模型正确。

其次根据软件激励设计方案，使用封装的API实现了对FFT电路的实现软件仿真模型，根据输出结果确定了软件仿真模型正确实现。最后在FPGA硬件上实现了软混合仿真，用软件仿真模型访问周期精确硬件模型实现并评估了仿真加速功能。

最后进行了混合仿真平台的功能验证，设计了QEMU仿真器执行的裸机程序，对APB SRAM电路进行随机读写，验证的通过说明了QEMU侧软件仿真模型与FPGA侧正常通信并且Adaptor层中信号转换模块正常实现，证明周期精确硬件模型正确地集成进入混合仿真平台，能够进行系统级别的仿真。

# 结论

本文中设计并实现周期精确硬件模型在FPGA上的仿真加速功能，能够以较低成本实现Verilog的仿真加速功能，一定程度上加速对IP核电路的开发速度。同时将FPGA加速的周期精确硬件模型集成到混合仿真平台中，使得FPGA仿真加速可以用于系统级仿真，进而加速对SoC的开发。

课题取得了以下的研究成果：

(1) 实现了周期精确硬件模型的自动化生成流程。在 Yosys 编译器工具中增加新的 Pass，在 Pass 中直接修改电路的 RLTL 中间表达形式来实现电路的 LI-BDN 化，并自动化实现相应的硬件接口来对周期精确硬件模型电路实现控制，可以通过执行相应的脚本来完成周期精确硬件模型的自动化实现，实现硬件仿真模型，并将硬件仿真模型在 FPGA 硬件上实现。
(2) 实现基于 Zynq 的嵌入式混合仿真平台，基于 Zynq 平台特点，实现软硬件通信，设计并实现了 Adaptor 中软件接口，根据软件接口提供的函数设计软件激励，实现混合仿真平台中的软件仿真模型。
(3) 实现基于 QEMU 的混合仿真平台, 在已有基于 Zynq 的混合仿真平台基础上, 在 Zynq 中增加通信模块, 与 QEMU 实现跨平台交互, 并在通信模块与软件接口间实现了信号转换模块, 实现事务级信号与引脚、周期精确级信号的互相转换。
(4) 完成各个部分的功能测试。针对 FFT 电路，对比编译器处理前后的电路结构，观察仿真模型的关键波形，确定编译器成功实现。然后根据 FFT 电路的周期精确硬件模型设计了相应的软件激励测试，通过终端输出确定软件激励工作正确性，并且验证仿真加速功能。最后验证混合仿真平台的实现，通过 QEMU 中的软件仿真模型对 APB 3.0 接口的 SRAM 电路访问，证明了基于 QEMU 的混合仿真平台顺利实现。

课题中实现了混合仿真平台中周期精确硬件模型的FPGA加速，但是仍有需要改进的部分。首先在编译器处理电路自动化生成方面，电路的LI-BDN化只对顶层模块进行处理，虽然满足设计需求，但是没有充分利用LI-BDN的优点，性能相比对多个模块LI-BDN化低。其次电路模型中的硬件接口可以实现更高性能的接口，以提高软硬件数据交互效率。在混合仿真平台开发中，受多方面限制使用了以太网作为通信方式，后续应使用PCIe等更高速的接口。
